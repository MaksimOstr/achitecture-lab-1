# Лабораторна робота №2 — Контейнеризація застосунків

## Що робилося

Проводились експерименти з пакуванням у Docker-образи трьох застосунків:

1. **Основний проєкт** — Java/Spring Boot Notes Service (цей репозиторій)
2. **Python-застосунок** — окремий застосунок на Python
3. **Go-застосунок** — окремий застосунок на Go

Для кожного застосунку збирались два образи:
- **не-оптимізований** (один шар/single-stage, без поділу на build і runtime)
- **оптимізований** (multi-stage build або зменшений базовий образ)

Вимірювались розмір фінального образу та час збірки (`docker build`).

---

## Результати

### Java/Spring Boot (основний проєкт)

| Варіант | Розмір образу | Час збірки |
|---|---|---|
| Single-stage (`Dockerfile.single-stage`) | ~1.5 GB | 79.2 с |
| Multi-stage (`Dockerfile`) | ~520 MB | 61.6 с |

**Конфігурація для відтворення:**
- Single-stage Dockerfile: [`Dockerfile.single-stage`](Dockerfile.single-stage)
- Multi-stage Dockerfile: [`Dockerfile`](Dockerfile)

```powershell
# single-stage
docker build -t mywebapp:single-stage -f Dockerfile.single-stage .

# multi-stage
docker build -t mywebapp:multistage -f Dockerfile .

docker images mywebapp
```

**Ключова різниця між Dockerfile'ами:**

Single-stage ([`Dockerfile.single-stage`](Dockerfile.single-stage)) залишає у фінальному образі весь `gradle:8-jdk21` (~830 MB) разом з вихідним кодом, Gradle cache і повним JDK.

Multi-stage ([`Dockerfile`](Dockerfile)) компілює у `gradle:8-jdk21`, а в продакшн-образ копіює лише зібраний `.jar` поверх `eclipse-temurin:21-jre` (~290 MB) — без JDK, без Gradle, без source code.

### Python

| Варіант | Розмір образу | Час збірки |
|---|---|---|
| Не-оптимізований | ~1.79 GB | 26.0 с |
| Оптимізований | ~238 MB | 11.9 с |

**Що зроблено для оптимізації:**
- Multi-stage build: стадія `builder` на `python:3.11-slim` встановлює пакети з `pip install --prefix=/install`, стадія `runtime` копіює лише готові пакети через `COPY --from=builder /install /usr/local` — без pip, без кешу, без build-залежностей
- `--no-cache-dir` у pip виключає кешування wheel-ів прямо під час встановлення
- `PYTHONDONTWRITEBYTECODE=1` і `PYTHONUNBUFFERED=1` прибирають `.pyc`-файли і буферизацію stdout — невеликий, але чистий виграш
- Базовий образ `python:3.11-slim` замість повного `python:3.11` — відсутні компілятори, заголовкові файли та більшість системних пакетів, які потрібні лише під час збірки

**Особливість:** через два `FROM python:3.11-slim` (builder і runtime) фінальний шар не містить pip і тимчасових артефактів встановлення — лише сам застосунок і встановлені пакети. Без multi-stage навіть `--no-cache-dir` не позбавляє образ від метаданих pip і проміжних файлів.

### Go

| Варіант | Розмір образу | Час збірки |
|---|---|---|
| Не-оптимізований | ~1.32 GB | 11.58 с |
| Оптимізований | ~27.7 MB | 9.44 с |

**Що зроблено для оптимізації:**
- Multi-stage build: компіляція у `golang:1.21-alpine`, runtime у `alpine:3.19` — Go toolchain (сам по собі ~300 MB) у фінальний образ не потрапляє
- Залежності (`go.mod` / `go.sum`) копіюються і завантажуються (`go mod download`) окремим кроком до копіювання source code — цей шар кешується і не перебудовується при зміні коду
- `CGO_ENABLED=0 GOOS=linux` — статично пов'язаний бінарник, сумісний із `alpine` (musl libc) без зовнішніх залежностей

**Особливість:** `alpine:3.19` (~7 MB) — не порожній `scratch`, але містить shell і базові системні утиліти, що зручно для діагностики. Go-бінарник при цьому статичний, тому libc в образі не потрібна — він просто запускається. Зменшення з 1.32 GB до 27.7 MB (~98%) досягається виключно за рахунок того, що Go toolchain лишається у builder-стадії.

---

## Труднощі та їх причини

**1. Розмір базових образів Java**

Навіть оптимізований multi-stage образ важить ~520 MB, бо JRE від Eclipse Temurin сам по собі великий. Можна ще зменшити через `jlink` (кастомний мінімальний JRE), але це суттєво ускладнює Dockerfile і виправдане лише у специфічних сценаріях.

**2. Час збірки Java**

Single-stage зайняв 79.2 с, multi-stage — 61.6 с (обидва виміряні з `--no-cache`). Різниця пояснюється тим, що multi-stage не виконує зайвих `apt-get` кроків у builder-стадії, а основний час (завантаження залежностей Gradle, компіляція, annotation processing Lombok) однаковий в обох варіантах. При повторній збірці зі збереженим кешем час різко падає — шар із завантаженими залежностями не перебудовується.

**3. Python і C-розширення на alpine**

`python:alpine` дає менший образ ніж `python:slim`, але частина пакетів (наприклад, `psycopg2`, `cryptography`) не має prebuilt wheel для musl libc і компілюється з нуля — потрібні `gcc`, `musl-dev` як build-залежності, а сам процес довший. При цьому навіть `python:alpine` не наближається до Go за розміром — 27.7 MB проти ~238 MB, бо Python-рантайм сам по собі важкий незалежно від базового образу.

**4. Go і CGO**

Якщо застосунок використовує CGO (наприклад, драйвер sqlite через cgo), статичний бінарник для `scratch` не збереться без додаткових налаштувань. `CGO_ENABLED=0` вимикає CGO і дозволяє зібрати повністю статичний бінарник.

## Висновки та рекомендації

### Що показують цифри

| Мова | Не-опт. | Оптим. | Зменшення |
|---|---|---|---|
| Java (Spring Boot) | 1.5 GB | 520 MB | ×2.9 |
| Python | 1.79 GB | 238 MB | ×7.5 |
| Go | 1.32 GB | 27.7 MB | ×47.7 |

Go демонструє найдраматичніше зменшення завдяки статичній компіляції та можливості використати `scratch`-образ. Python і Java навіть після оптимізації залишаються відносно великими — їм потрібен рантайм у контейнері.

### Рекомендації

1. **Завжди використовуйте multi-stage build** для мов зі стадією компіляції (Java, Go, Rust, C#). Зберігати build-toolchain у продакшн-образі немає жодного сенсу.

2. **Вибирайте мінімальний базовий образ для runtime.** Для JVM — `eclipse-temurin:XX-jre` замість `-jdk`. Для Python — `python:X.X-slim` як безпечний вибір за замовчуванням, `alpine` — тільки якщо ви впевнені у сумісності залежностей.

3. **Порядок шарів у Dockerfile має значення.** Копіюйте файли залежностей (`pom.xml`, `requirements.txt`, `go.mod`) і встановлюйте їх окремим кроком *до* копіювання source code — тоді цей шар кешується і не перебудовується при кожній зміні коду.

4. **Менший образ — менша поверхня атаки.** `scratch`/`distroless` не містять shell і системних утиліт, що унеможливлює цілий клас атак.

5. **Для Java розгляньте `jlink`** якщо розмір образу критичний — можна зменшити до ~100–150 MB за рахунок включення лише потрібних JDK модулів. Ціна — складніший Dockerfile.

6. **Не оптимізуйте передчасно.** Для локальної розробки та CI зручніший повний образ з усіма інструментами. Оптимізований образ потрібен у продакшн і/або при великій кількості деплойментів, де розмір впливає на швидкість pull і використання місця у registry.
